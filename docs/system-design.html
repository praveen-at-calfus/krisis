How a raw ticket becomes a routed decision

The core idea: let the model judge, let code decide. The LLM reads the ticket and assesses it; deterministic code turns that assessment into the parts that must never drift — priority and team.

Inside POST /classify

1
Validate input guard
Empty or >20k chars → clean 422. Over-long input is truncated — never a crash.
2
Embed the ticket OpenAI
One vector, reused for both the cache check and the similar-tickets panel.
3
Semantic cache lookup Postgres
Cosine-match past tickets. If ≥ 0.92 similar → reuse the prior answer and skip the LLM (time + tokens saved).
4
Classify with the LLM judgment OpenAI
Structured output — the model reasons then commits: analysis → impact → urgency → category → reasoning → confidence. Reliability: enforce schema → retry with the error fed back → safe fallback.
5
Derive in code deterministic
priority = Impact × Urgency matrix (+ overrides) · assigned_team = taxonomy lookup · needs_review = low confidence or unclassified.
6
Log everything Postgres
Answer + embedding + tokens/cost + cache provenance + confidence — powering the dashboard, cache, and incident alarm.

↳ returns { category, priority, assigned_team, reasoning, confidence, needs_review }

Why the split matters

LLM decides (judgment)

  • category — which of 7 buckets
  • impact — broad vs narrow
  • urgency — blocked vs workaround
  • reasoning + confidence

Code decides (deterministic)

  • priority — from the matrix, never guessed
  • assigned_team — from the category
  • needs_review — from confidence
  • → consistent, defensible, can't mismatch

Priority = Impact × Urgency

blockedworkaround
broadHighMedium
narrowMediumLow

Overrides applied after the matrix: security → High · unclassified → Medium. Priority follows impact & blockage, not tone — an angry ticket never outranks a calm, more severe one.

Capabilities layered around the core

Semantic cache

Near-duplicate tickets reuse a prior answer and skip the LLM.

Retrieval (RAG)

Shows similar past resolved tickets and how they were fixed.

Confidence-aware routing

Low-confidence tickets flagged for human review, not auto-trusted.

Incident clustering

A same-category spike raises a dismissible dashboard alarm.

Cost tracking

Per-ticket $ and cache savings, on the dashboard.

Reliability

Retry → safe fallback; the API never crashes on a bad response.
